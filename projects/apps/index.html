Introdução

Visão Geral

Entenda como funciona o processo de build e publicação de novos apps mobile multi-tenant.

Sobre este guia

Este guia descreve o processo completo para criar e publicar um novo app nas lojas (App Store e Google Play) a partir do repositório interno react-native-app.

A plataforma é multi-tenant: o mesmo código gera apps diferentes para cada cliente, diferenciados pelo bundle ID e pelo nome do app. Cada tenant gera um app independente nas lojas — com ícone, nome e pacote próprios.

ℹ️

O bundle ID de cada tenant segue o padrão com.plataformaexemplo.slug-do-tenant. O slug é definido no momento de criar o tenant.

Fluxo geral

O processo é dividido em três fases:

1. Pré-build — configuração de ambiente antes de gerar os binários:

  • Adição dos assets do tenant (ícone, splash)
  • Build iOS local com eas build --platform ios (interativo, uma vez por tenant)
  • Configuração do Firebase Android (google-services.json)
  • Disparo do workflow Android no GitHub Actions

2. Pós-build — configuração de push notifications após os builds estarem prontos:

  • Criação do aplicativo de plataforma iOS no Amazon SNS (APNs)
  • Criação do aplicativo de plataforma Android no Amazon SNS (FCM)
  • Registro dos ARNs no GI da plataforma

3. Publicação — envio dos apps para as lojas:

  • Publicação na App Store Connect
  • Publicação no Google Play Console

O que o workflow faz

O workflow Android executa automaticamente:

  1. Faz bump da versão do app (patch, minor ou major)
  2. Commita e faz push da nova versão no repositório
  3. Define as variáveis do tenant no EAS (nome e slug)
  4. Inicia o build Android no EAS (assíncrono)
ℹ️

O build iOS é feito localmente antes de disparar o workflow — consulte a seção Configuração iOS.

✅

O build Android roda no EAS em background. Acompanhe no painel de builds do projeto no expo.dev após disparar o workflow.

Novo Tenant

Assets do Tenant

Imagens de ícone e logo obrigatórias antes do primeiro build. Sem elas, o build falhará.

⚠️

O build falhará se os assets do tenant não existirem no repositório antes de disparar o workflow. Este passo é obrigatório.

Arquivos obrigatórios

Cada tenant precisa de exatamente dois arquivos dentro da sua pasta:

ArquivoUso
logo.png Logo exibida dentro do app (splash screen, telas de auth, etc.)
adaptive-icon.png Ícone do app no Android (adaptive icon — camada foreground). Também usado como ícone no iOS.
💡

Use o template do Figma Community para exportar os assets nos tamanhos corretos: Expo App Icon & Splash (v2) — Figma Community.

Especificações

ArquivoTamanho recomendadoFundoFormato
logo.png Mínimo 512 × 512 px Transparente ou sólido PNG
adaptive-icon.png 1024 × 1024 px Transparente — o fundo é definido pelo tema do app PNG
adaptive-icon.png
logo.png
ℹ️

O adaptive-icon.png deve ter fundo transparente. O Android aplica a cor de fundo do app por baixo — a imagem deve conter apenas o foreground (logo/ícone).

⚠️

Sempre rode o app localmente antes de publicar. Verifique se o ícone e a splash screen estão com proporções corretas e bem centralizados — tanto no Android quanto no iOS. Problemas de alinhamento ou corte só ficam visíveis rodando o app de verdade.

Onde colocar

Os arquivos devem estar em:

assets/images/tenants/slug-do-tenant/
├── logo.png
└── adaptive-icon.png

Commite e faça push dos arquivos antes de disparar o workflow. Exemplo para o tenant minha-empresa:

git add assets/images/tenants/minha-empresa/
git commit -m "feat: add tenant assets for minha-empresa"
git push
💡

Consulte a pasta de algum tenant já existente em assets/images/tenants/ como referência de estrutura e nomes de arquivo.

Novo Tenant

Configuração iOS

O build iOS é feito localmente — rode o comando abaixo antes de disparar o workflow Android.

Slug do tenant

Digite o slug para preencher automaticamente os bundle IDs e comandos desta página.

com.plataformaexemplo.

Por que é local

Cada tenant gera um app com bundle ID próprio — por exemplo, com.plataformaexemplo.minha-empresa. A Apple exige um Provisioning Profile exclusivo por bundle ID, e o EAS só consegue criá-lo no primeiro build em modo interativo.

Por isso o build iOS é feito localmente uma vez por tenant. Builds futuros do iOS também precisam ser feitos localmente — não há workflow automático para iOS.

ℹ️

O Distribution Certificate é compartilhado entre todos os tenants da mesma conta Apple. Apenas o Provisioning Profile é criado por tenant.

Passo a passo

⚠️

Durante o processo, o EAS solicitará login com a conta Apple Developer. Use a mesma conta configurada nos pré-requisitos.

1
Clone o repositório e instale as dependências
git clone <URL_DO_REPOSITÓRIO>
cd react-native-app
yarn install
2
Crie o perfil no eas.json e rode o build

Adicione o perfil do tenant ao objeto "build" do eas.json e commite antes de buildar:

TENANTNAME= 
"production-slug-do-tenant": {
  "autoIncrement": false,
  "node": "24.0.0",
  "environment": "production",
  "env": {
    "EXPO_PUBLIC_TENANT": "slug-do-tenant",
    "TENANTNAME": "Nome do Tenant"
  }
}

Em seguida, rode o build:

eas build --platform ios --profile production-slug-do-tenant
💡

O EAS solicitará credenciais interativamente na primeira vez — confirme a criação do Provisioning Profile quando perguntado.

⚠️

Se o EAS perguntar se deseja recusar o Distribution Certificate existente, não recuse — ele é compartilhado entre todos os tenants. Apenas o Provisioning Profile deve ser gerado novo.

3
Aguarde o build e submeta para a App Store

O build roda no EAS. Acompanhe no painel de builds do projeto em expo.dev → Builds. Ao concluir, configure as credenciais do tenant e submeta.

Primeiro, vincule a API Key ao tenant correto:

eas credentials --platform ios --profile production-slug-do-tenant

No menu, selecione App Store Connect: Manage your API Key → Use an existing API Key for EAS Submit → selecione a key mais recente.

Em seguida, rode o submit:

eas submit --platform ios --profile production-slug-do-tenant

O EAS exibirá um menu de opções — selecione Select a build from EAS e depois escolha o build mais recente da lista.

4
Prossiga para a Configuração Android

Com o build iOS em andamento, configure o Firebase e dispare o workflow Android. Consulte a seção Configuração Android.

Novo Tenant

Configuração Android

Setup único necessário antes do primeiro build Android de cada novo tenant.

Slug do tenant

Digite o slug para preencher automaticamente os bundle IDs e comandos desta página.

com.plataformaexemplo.

Por que é necessário

O arquivo google-services.json lista os package names autorizados no Firebase. Cada tenant possui um package name único — por exemplo, com.plataformaexemplo.minha-empresa.

Se o package name do novo tenant não estiver no arquivo, o build falhará com o erro:

No matching client found for package name 'com.plataformaexemplo.slug-do-tenant'
in /home/expo/workingdir/build/android/app/google-services.json

Registrar no Firebase

1
Acesse o Firebase Console

Entre no Firebase Console e selecione o projeto interno de notificações.

2
Clique em "+ Adicionar app"

Na visão geral do projeto, clique no botão + Adicionar app.

3
Selecione a plataforma Android

Nas opções de plataforma exibidas, clique no ícone Android.

4
Informe o Package name

No campo Nome do pacote do Android, informe o slug do tenant — o prefixo é sempre fixo:

com.plataformaexemplo.

Baixar google-services.json

Após registrar o app no Firebase, baixe o arquivo google-services.json atualizado e substitua o existente na raiz do projeto.

5
Baixe o google-services.json

No Firebase Console, após adicionar o app Android, clique em Baixar google-services.json. O arquivo já conterá todos os apps Android registrados no projeto.

💡

Após baixar o arquivo, clique em Avançar nos passos seguintes e depois em Voltar ao console — não é necessário seguir as instruções de integração do SDK.

6
Substitua o arquivo no repositório

Copie o arquivo baixado para a raiz do projeto, substituindo o google-services.json existente. Commite e faça push. Em seguida, prossiga para a seção Executar o Workflow.

⚠️

Sempre baixe o arquivo diretamente do Firebase após registrar um novo tenant. Não edite o JSON manualmente.

CI/CD

Executar o Workflow

Como disparar o build Android de um tenant pelo GitHub Actions. O build iOS é feito separadamente — veja Configuração iOS.

Acessar o GitHub Actions

1
Abra o repositório no GitHub

Acesse o repositório interno no GitHub e clique na aba Actions.

2
Selecione o workflow

Na lista à esquerda, clique em Start android build in EAS.

3
Clique em "Run workflow"

Um formulário aparecerá para preencher os parâmetros do build. Preencha e confirme.

Parâmetros

CampoDescriçãoExemplo
tenant_name Nome de exibição do app nas lojas Minha Empresa
expo_public_tenant Slug do tenant — define o bundle ID e as configurações carregadas. Deve ser idêntico ao usado no setup iOS e Firebase minha-empresa
bump_type Tipo de bump de versão. Use patch para builds de rotina patch
bundle_id Prefixo do bundle ID. Para apps com o padrão atual, use com.plataformaexemplo. Para apps antigos com bundle ID customizado, informe o prefixo correto — o slug do tenant é adicionado automaticamente pelo workflow com.plataformaexemplo
💡

O expo_public_tenant deve ser exatamente o mesmo slug usado no setup iOS e no registro do Firebase.

Monitorar o build

O build roda de forma assíncrona no EAS. Para acompanhar:

  1. Acesse o painel de builds do projeto em expo.dev → Builds
  2. O build aparecerá com status In progress
  3. Ao concluir, o artefato .aab fica disponível para download
💡

Com o build concluído, configure as notificações push antes de publicar nas lojas — veja Configuração SNS iOS e SNS Android.

Pós-publicação

Pós-publicação iOS

Passos a realizar após o app iOS ser aprovado na App Store.

Registrar App ID no eas.json

O ascAppId é o ID do Apple gerado pela App Store Connect ao criar o app. Ele é necessário para que o eas submit saiba para qual app enviar o build nas próximas atualizações.

1
Localize o ID do Apple no App Store Connect

Acesse appstoreconnect.apple.com → Apps, abra o app do tenant e clique em Informações do app no menu lateral. Role até a seção Informações gerais e copie o valor do campo ID do Apple (número com ~10 dígitos, ex: 123456789).

2
Adicione o ascAppId no eas.json

No objeto "submit" do eas.json, adicione o campo ascAppId ao perfil do tenant:

"production-slug-do-tenant": {
  "ios": {
    "ascAppId": "1234567890"
  }
}

Commite e faça push da alteração.

💡

Com o ascAppId preenchido, o comando eas submit nas próximas atualizações não precisará de confirmação manual do app de destino.

Atualizar iOS

Como gerar novo build iOS e submeter uma atualização para um tenant existente.

⚠️

Tenants mais antigos podem não seguir o padrão atual de bundle ID (com.plataformaexemplo.[slug-do-tenant]). Sempre confirme o bundle ID correto consultando o painel de apps publicados na App Store Connect antes de gerar o build.

Se o app já estiver publicado com um bundle ID diferente do padrão, use a opção App existente abaixo para informar o prefixo correto.

Tipo de app
Slug do tenant

Digite o slug e o nome para preencher automaticamente os comandos desta página.

com.plataformaexemplo.
TENANTNAME=
bundle_id=
ℹ️

Todas as alterações (código, assets, configurações) precisam estar commitadas e no repositório remoto antes de gerar o build — o EAS usa o estado atual do branch.

ℹ️

O workflow Android bumpa runtimeVersion e versionCode automaticamente via scripts/bump-version.js. Para manter iOS e Android na mesma versão, dispare o workflow Android primeiro, faça git pull do commit gerado e só então rode o build iOS.

⚠️

O build iOS é sempre manual — não há workflow automático para iOS.

Perfil no eas.json

Confirme se o tenant tem perfis em eas.json. Os nomes seguem o padrão production-[slug]. Se não existirem, adicione os dois trechos abaixo ("build" e "submit") e commite antes de buildar.

No objeto "build":

"production-slug-do-tenant": {
  "autoIncrement": false,
  "node": "24.0.0",
  "environment": "production",
  "env": {
    "EXPO_PUBLIC_TENANT": "slug-do-tenant",
    "TENANTNAME": "Nome do Tenant"
  }
}

No objeto "submit":

"production-slug-do-tenant": {}

Para apps com bundle ID customizado, inclua também o campo BUNDLE_ID no perfil de build — assim os comandos de build e submit dispensam variáveis de ambiente inline:

No objeto "build":

"production-slug-do-tenant": {
  "autoIncrement": false,
  "node": "24.0.0",
  "environment": "production",
  "env": {
    "EXPO_PUBLIC_TENANT": "slug-do-tenant",
    "TENANTNAME": "Nome do Tenant",
    "BUNDLE_ID": "com.prefixo-existente.slug-do-tenant"
  }
}

No objeto "submit":

"production-slug-do-tenant": {}
